# AI 对话知识库

从 Cursor、Claude Code、claude-internal、CodeBuddy（插件 + IDE）收集对话，筛选出值得沉淀的内容，提炼成可复用的个人知识笔记。

这个 SKILL 只负责三件事：

1. 收集并浏览对话归档
2. 选择值得提炼的对话
3. 生成笔记并更新索引状态

详细的数据源格式见 `references/data-sources.md`；命令和参数细节见 `references/cli-commands.md`；风格样例见 `references/example-notes/`。

## 先做什么

### 1. 找到脚本

CLI 在同目录 `scripts/dialogue-kb.py`。由于 SKILL 可能通过符号链接分发，先用 Glob 找到真实路径，再用绝对路径调用。后文用 `$KB` 代指该脚本绝对路径。

```text
**/skills/dialogue-knowledge/scripts/dialogue-kb.py
```

### 2. 读取记忆

记忆文件是同目录下的 `memory.md`。若存在，先读取并遵循其中的长期偏好；若不存在，按默认流程执行。

```text
**/skills/dialogue-knowledge/memory.md
```

`memory.md` 适合记录这些长期偏好：

- 默认存储通道
- 启用或禁用哪些数据源
- 提炼时每次处理多少条
- 语言、写作风格、抽象层级
- 默认脱敏策略

当用户表达了稳定偏好时，主动提议更新 `memory.md`。更新时使用当前环境提供的文件编辑工具，不要依赖手动编辑 `index.json`。

## 交互边界

这个 SKILL 默认采用**人参与（human-in-the-loop）**模式，因为它涉及"替用户决定什么值得记住"。

默认规则是：**读和查可以自治，写和定夺必须门控。**

只有在以下两种情况下，Agent 才可以放宽某一步的门控：

- 用户在当前消息中明确授权，例如"你直接挑 3 条处理"、"以后先生成并直接存 local"
- `memory.md` 中已经记录了明确、稳定的长期偏好

即使放宽了门控，也应在结果里说明你替用户做了什么决定。

### 可以自治执行（不需要等用户）

- `collect` / `scan` / `index` — 纯数据操作，不改变任何知识内容
- `triage` — 只是读取摘要，不做任何修改
- 对摘要中**明确无技术价值**的对话批量 `skip` — 闲聊、空壳、明显未完成、纯命令、无回复等。如果摘要里仍有技术主题可能性，就不要替用户跳过，放进候选列表。

### 必须暂停等用户确认（硬性门控）

默认情况下，以下三个位置，Agent **必须停下来展示信息并等待用户回复**，不得自行继续；除非用户当前明确授权，或 `memory.md` 已经为这一步给出长期授权。

1. **选择对话** — triage 之后，把候选列表展示给用户，让用户挑选要处理哪些。不要默认替用户决定"这几条最值得看"然后直接全处理掉。
2. **审阅笔记** — 生成笔记草稿后，把完整内容展示给用户看。用户可能要求修改标题、调整内容、更换风格、补充细节，或者决定不要这篇笔记。**在用户明确同意之前，不得写入任何存储通道。**
3. **选择通道** — 如果记忆中已有默认通道，可以沿用；如果没有默认通道，就把 `local` 作为建议默认值，但仍要在审阅时明确告知"将存到 local，若你想换通道请告诉我"，并等待用户确认后再写入。

### 判断原则

当你不确定是否需要暂停时，用这个简单标准：**如果这一步的结果对用户来说是不可逆的、难以察觉的，或带有主观取舍（写了文件、写入外部系统、标记 done、替用户淘汰候选），就必须先让用户看到并确认。** 纯读取和纯查询操作可以自治。

## 默认工作流

当用户第一次使用，或只说"帮我整理对话"但没指定细节时：

1. 读取 `memory.md`
2. 运行 `$KB collect`
3. 运行 `$KB triage`
4. 展示候选列表给用户，**等用户选择** ← 门控 ①
5. 用户选定后，对每条运行 `$KB show`，阅读全文
6. 生成笔记草稿，**展示给用户审阅** ← 门控 ②
7. 若记忆中没有默认通道，把 `local` 作为建议默认值一并告知；用户确认后，写入存储通道并运行 `$KB done`
8. 若是首次使用且没有 `memory.md`，在结束时引导用户建立偏好

一次处理 1-3 条即可，不必追求清空全部 backlog。

## 如何判断用户意图

优先把请求归到以下四类：

| 意图 | 常见表达 | 动作 |
|------|----------|------|
| 收集 | "收集对话" "同步一下" "扫描下" | `scan` / `collect` |
| 提炼 | "整理对话" "总结一下" "生成笔记" | `triage` → `show` → 提炼 |
| 浏览 | "列出来看看" "搜一下" | `list` / `show` |
| 统计 | "概览" "有多少条" | `stats` |

不明确时先问一句，不要直接长链路执行所有动作。

## 收集与浏览

当你需要确认命令参数、远程行为、分页、筛选或状态字段时，再读取 `references/cli-commands.md`。如果你只是在决定"下一步该执行哪个动作"，不必加载这份参考文档。

### 收集原则

- `collect` 和 `scan` 默认处理本机，并尝试 `~/.ssh/config` 中的远程主机
- 若用户明确只看本机，使用 `--local-only`
- 若用户只想看特定远程，使用 `--remote HOST...`
- 远程扫描可能慢，也可能失败；向用户说明范围和结果，不要把失败误报成"没有新内容"

常用命令：

```bash
$KB scan -v
$KB collect -v
$KB collect --local-only
$KB collect --remote $REMOTE_HOST
```

### 浏览原则

只在需要时展开全文。优先用摘要做第一轮筛选，再用全文做第二轮判断。

```bash
$KB list --pending
$KB triage
$KB show <ID>
$KB show <ID> --full
$KB stats
```

`triage` 输出的字段足够做初筛：`id`、`title`、`first_question`、`turns`、`tool`、`host`、`project`、`state`。

## 提炼流程

采用三层漏斗：

```text
Layer 0  脚本自动过滤  -> 太短 / 无回答 / 纯命令
Layer 1  AI 读摘要判值 -> 决定哪些值得深读
Layer 2  AI 读全文提炼 -> 只处理少量高价值对话
```

### 1. 先用 `triage` 批量判值

拿到 triage 结果后，把对话分成三类：

- 值得提炼：有问题、有过程、有结论
- 可能有价值：标题或问题像技术话题，但需要看全文确认
- 直接跳过：闲聊、极短、纯命令、明显未完成

能确定无价值的可以批量 `skip`，其余交给用户选择。

### 2. 再用 `show` 深读全文

阅读时重点关注：

- 用户原始问题是什么
- 最终有没有解决
- 关键转折点是什么
- 是否形成了可复用知识

对话很长时先看默认截断版，只有在确实需要时再用 `--full`。

### 3. 生成笔记

默认写作原则：

- 开头 1-2 句话让未来的自己立刻知道这篇讲什么
- 只保留难记、反直觉、容易踩坑的部分
- 不机械复述整个对话过程
- 一条对话生成一篇笔记
- 代码只保留关键片段，不做大段抄录

风格优先级：

1. 当前用户的明确要求
2. `memory.md` 中的长期偏好
3. 本节默认原则

若用户偏好"通用化"或"默认脱敏"，主动把项目名、内部系统名、目录路径、人员信息改写成通用表达。

参考风格时，按需读取：

- `references/example-notes/debug-example.md`
- `references/example-notes/research-example.md`
- `references/example-notes/implementation-example.md`

### 4. 存储笔记

存储采用 channel 思路。

默认只保证 `local` 通道可用；其他通道都应视为用户自定义扩展，并记录在 `memory.md` 中。只有在记忆里明确声明了某个通道及其写入方式时，才应直接使用；否则先询问用户。

当前默认约定如下：

| 通道 | 说明 |
|------|------|
| `local` | 默认通道，写入 `~/.ai-dialogues/notes/` 下的 Markdown |

若记忆中已有默认通道，直接沿用；否则把 `local` 作为建议默认通道，并在审阅笔记时明确告知，等用户确认后再存储。如果用户希望同步到其他地方，先确认通道名称和写入方式，再写入 `memory.md`。

本地笔记建议保留如下 frontmatter：

```yaml
title: <具体标题>
type: debug | research | implementation | optimization | learning
category: Frontend | Backend | DevOps | Database | Architecture | Tools | Life | Other
tags: [具体技术标签, 2-5 个]
source_tool: cursor | claude | codebuddy | codebuddy-ide
source_id: <对话ID>
created: YYYY-MM-DD
```

### 5. 更新状态

笔记写完后，必须通过 CLI 更新状态，不要手动改 `index.json`。`done --channels` 只记录通道元数据，不负责实际写入：

```bash
$KB done <ID...> --channels "local" --note-title "笔记标题"
$KB skip <ID...> --reason "无实质内容"
$KB reset <ID...>
```

若已提炼过的对话后续又增长，列表中会显示 `↑N`。这类对话应重新阅读新增部分，再决定是更新笔记还是仅重新标记为 done。

## 关键原则

- **不可逆或主观取舍的操作必须先确认** — 写文件、写入外部系统、标记 done、替用户淘汰候选前，必须让用户看到内容并同意
- 先读记忆，再执行动作
- 先用摘要筛，再读全文
- 让用户决定优先处理哪些对话
- 每次少量处理，优先保证质量
- 用命令更新状态，不手改索引
- 长说明放在 `references/`，正文只保留决策规则和主流程
