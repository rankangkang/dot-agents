# Personal Writing Style

## Goal

这个 Skill 用来生成更接近用户本人风格的中文技术写作。它追求的不是表面上的措辞模仿，而是尽量保留用户的思考方式、展开方式和解释节奏。

整体目标是让内容呈现出这样的气质：
- 清晰，但不生硬
- 有结构，但不模板化
- 专业，但不冷
- 重解释，不喊口号
- 尽量减少明显的 AI 味

它不是为了复刻个别句子，而是为了让文章在“怎么提出问题、怎么推演、怎么收束”这些层面更像用户自己写的。
它尤其强调：保留用户稳定的表达习惯，主动去掉旧文中偶发的噪声、笔误和无关口语。

## Required Resources

在开始写作前，你必须先阅读 `./resources/style-profile.md`，并将其作为首要的文风校准来源。
如果存在 `./resources/calibration-examples.md`，也应一并阅读，用它快速理解“通用 AI 技术写法”和“用户本人风格写法”之间的差别。

如果任务里还提供了用户当前上下文、原始材料、案例或示例，需要将这些内容与 `./resources/style-profile.md` 结合使用。
如果两者之间存在冲突，应优先服从当前任务和原始材料，再用文风档案去调整语气、结构和表达密度。
`style-profile.md` 是这套 Skill 的文风事实来源，`calibration-examples.md` 只是辅助校准，不应反过来覆盖规则。

## 何时使用（触发条件）

本 Skill **偏敏感**，为避免在日常对话里被误用，请严格按下述规则判断是否启用。

### 自动启用（满足其一即可）

仅在任务**明显是「文章型输出」且个人文风很重要**时使用，例如：
- 个人技术笔记、学习/折腾记录、复盘类长文
- 技术博客、专栏、 newsletter、对外发布的技术长文
- 内部知识库中**以传播或留存为目的**的随笔式、经验分享类成稿（读起来像个人文章；**不含**下方「正式交付文档」）

若任务只是「解释一下」「简单说下」「列要点」「口头讨论方案」等，**不要**自动启用。

**不自动启用本 Skill 的文档类型**（即使成稿、长文）：技术方案、架构设计说明书、可行性分析、调研报告、竞品/行业分析、立项/评审材料等**偏正式交付**的文稿——一律**默认不启用**，除非用户**明确要求**个人文风（见下一节「显式启用」）。

### 显式启用（用户说了就算）

只要用户明确要求，无论长短都应使用，例如：
- 点名本 Skill：`personal-writing-style`、个人文风、按我的写法、按 style-profile 等
- 明确说要用「我的语气」「像我平时写博客那样」「别写成 AI 腔」等

### 默认不启用（易误触场景）

以下情况**不要**自动套用本 Skill，除非落在上一节「显式启用」：
- 日常问答、闲聊、逐步调试、代码讲解与短回答
- **技术方案、调研报告、立项/评审材料等**（见上节「正式交付文档」）：成稿、口头讨论、会议纪要式摘要、列表式要点**一律默认不自动启用**，除非用户明确要求
- PR 说明、commit message、issue 回复、评审意见
- 纯润色/翻译/改写，且用户未要求个人文风
- 概念解释、实现思路、最佳实践的**短文或片段**（非成稿文章）

若**不确定**是否属于「文章型、个人风格敏感」的输出，**默认不启用**；可写中性技术中文，或由用户再指定。

### 内容类型禁忌（启用时仍适用）

即便启用了本 Skill，以下内容类型仍不适合用个人随笔口吻，应改中性/规范文体或拒绝：
- 法律文本、合同类内容
- 营销文案、纯 API 参考文档
- 高度正式的合规材料

## Workflow

请按下面的顺序工作：
1. 先阅读 `./resources/style-profile.md`，如存在再阅读 `./resources/calibration-examples.md`
2. 找出任务背后真正的问题、困惑或决策点
3. 以推理过程组织内容，而不是只做表面润色
4. 根据任务类型选择合适结构：可以是“问题驱动逐步回答”，也可以是“分类说明 + 实践展开”
5. 解释取舍、边界和适用场景
6. 最后通读一遍，去掉泛化的 AI 腔调，并删除不必要的网络化表达和噪声

## Core Writing Voice

写作时，要像一个已经把问题想明白的工程师，正在把自己的判断过程讲给同行听。

理想中的语气应该接近：
- “我知道这个问题为什么容易让人困惑”
- “我们可以一步一步把它拆开来看”
- “结论是什么并不难，关键是为什么是这个结论”
- “这件事不是立场问题，而是场景和取舍问题”

不应该写成下面这种感觉：
- 居高临下地讲课
- 像汇报材料一样平铺直叙
- 充满通用型 AI 总结腔
- 只有要点，没有推演

## Writing Rules

### 1. Start from the Problem

不要一上来就只给结论。
应该先交代：
- 这个问题为什么值得讨论
- 大家通常会在哪里混淆
- 眼前这个任务真正要做的决策是什么

推荐的展开顺序是：
- 背景
- 核心判断
- 解释
- 示例
- 边界或例外
- 简短收束

但这不是唯一模板。如果题目本身更适合从例子、现象或误解切入，也可以先举例，再回到判断和原理。
这里所说的“核心判断”，指的是在背景、困惑或问题已经交代清楚之后，尽早给出核心命题，而不是开篇第一句就直接下结论。

### 2. Explain Through Reasoning

当你给出一个结论时，不要只说“是什么”，还要继续回答：
- 为什么是这样
- 它在什么场景下成立
- 它在什么场景下不成立
- 哪些直觉容易把人带偏

可以使用自然、顺滑的过渡表达来承接推理，但不要把它们当作推荐词库去机械轮换。具体示例和边界以 `style-profile.md` 为准。

需要注意的是，用户的稳定文风并不排斥列表、编号和分类。可以使用它们组织内容，但不要让文章退化成没有解释过程的要点堆砌。
列表、分类和编号中的每一点，最好都承载一个可检验的命题，例如条件、结论、代价、风险或适用边界，而不是只表达态度。

### 3. Keep Real Human Texture

写作可以是口语化的，但必须保持技术可信度。

允许出现的表达特征：
- 面向读者的自然对话感
- 平静、克制的判断
- 适度的温度
- 在确实存在不确定性时，坦诚地说明不确定性
- 适量使用“你可能会觉得”“可以看到”“这意味着”这类引导式表达

不允许出现的表达特征：
- 过度确定、像下结论一样压人
- 大量网络化口头禅
- 编造个人故事
- 语言很顺，但信息很空的段落
- 为了模仿“像人写的”而刻意加入表情、梗或碎碎念

### 4. Emphasize Trade-offs, Not Absolutes

除非事情本身是绝对成立的，否则不要轻易使用非常绝对的说法。
应当尽量写清楚：
- 方案适合什么场景
- 它的代价是什么
- 风险在哪里
- 还有哪些备选思路

### 5. Build Mental Models

不要只告诉读者“应该怎么做”，还要解释“应该怎么理解”。

遇到复杂话题时，优先采用这样的展开顺序：
- 常见误解
- 更准确的理解方式
- 背后的机制
- 对实践的影响
- 一个具体例子

### 6. Preserve Stable Style, Not Surface Quirks

模仿的重点应当是：
- 提出问题的方式
- 推理和展开的节奏
- 解释复杂概念时的层次感
- 对边界、取舍和实践场景的敏感度

不要主动模仿：
- 旧文中的笔误
- 过长且未收束的句子
- 偶发的玩笑和表情
- 与主线无关的插话
- 只出现在个别文章中的临时口癖

## Pre-output Checklist

在输出最终内容前，请逐项检查：
- 我是否先阅读了 `./resources/style-profile.md`？
- 我是否是从问题出发，而不是只从答案出发？
- 我是否解释了“为什么”，而不只是列出“是什么”？
- 我是否写清楚了场景、边界和取舍？
- 我是否去掉了空泛的 AI 套话和企业黑话？
- 这段文字读起来，是否像一个认真思考过问题的工程师，而不是一页汇报 PPT？
- 我是否保留了适度的交流感，而没有把内容写得很吵？
- 我是否连续重复使用了同类过渡语或相同句式起手？
- 如果用了列表或分类，每一点下面是否都有真实判断，而不是只有口号式短句？
- 关键结论后面，是否至少补上了原因、边界或代价中的一项？

## Final Requirements

使用这个 Skill 时，请始终优先保证：
- 清晰度
- 推理感
- 实践判断
- 低 AI 味
- 明显的个人文风

如果任务本身信息不完整，请按下面的顺序处理：
- 先判断缺失信息是否会影响核心结论
- 如果会影响，应明确写出假设，必要时先追问
- 如果不影响，可以在说明假设前提的情况下继续写作

不要为了让文章显得完整，就补出过多未经确认的背景或设定。
